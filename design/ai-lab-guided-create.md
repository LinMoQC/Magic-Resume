# AI 实验室 ·「引导创建」流程重塑 — 设计简报 v1.2

> 状态:草案 v1.2,方向已确认(2026-06-25)。本稿只定义 UX / 交互 / 状态与文案,不含实现代码。
> v1.1 增补:对话级「思考中 / 等待首响」状态与超时兜底(用户反馈:等待时没有任何状态消息)。见 §0.3 / §5 / §6 / §7 / §8.5。
> v1.2 决议(用户反馈 2026-06-25):① **读简历需授权**——采用「工具授权弹层」(Claude Code 式 human-in-the-loop),开场不再自动读,见新增 §6.5;② **无固定步骤**——进度由 AI 自主规划,删掉 `N/5` 固定步思维,见 §4 / §9;③ §8.1(带授权的开场)与 §8.2(右栏舞台)均**需 Magic-Core 后端参与**,属跨仓,见 §8 / §12。
> 关系:是 `ai-lab-living-canvas.md` 的姊妹篇。Living Canvas 解决"已有简历就地改";本稿解决"从零长出一份简历"——补上实验室里**唯一没有舞台**的场景。
> 设计上下文见根目录 `.impeccable.md`。

---

## 0. 现状与问题

入口:AI 实验室 → 技能「引导创建(create)」。

三个具体问题(用户反馈):

1. **第一句话写死**——`selectSkill` 的 `isChat` 分支直接 `addAssistant('好,我们从零开始。先告诉我:你的目标岗位和所在行业是什么?')`:字面量、每次一样、连 i18n 都没走。
2. **太单调**——选完技能后,只有一句气泡飘在空画布上,接着是无结构的纯聊天。根因:实验室北极星是「画布是舞台,对话是旁白」,而 create 是唯一**没有舞台**的场景(从零还没有简历可展示),于是整段体验只剩"旁白"。对照启动页(滚动示例 + 技能卡)反而更丰富,落差全在这里。
3. **等待无反馈**——从"发送"到 AI 吐出第一个字之间,对话流里没有任何"思考中"。`runChat` 的 assistant 气泡是**首个 `message_chunk` 才懒创建**的,空窗期只有 composer 变灰。analyze / optimize 走 `exec` 卡片有转圈,**唯独 chat(create / general)这条路没有等待态**。一旦开场改成动态生成(问题 1 的修法),这个空窗会从"看不见"变成"对着空画布干等",更刺眼——所以**等待态是动态开场的前置条件**,不是附带项。

设计意图本就更高:`AI能力使用手册` §2 写的是"边聊边**查看右侧预览**",且 `AiChatShell.runChat` 已能在流里接住 `resume_update / resume_patch` 草稿——**管线已通,只是 UI 没做到位**。

---

## 1. Feature Summary

把「引导创建」从"写死问候 + 无结构纯聊天"升级为"边聊边长出简历"的协作流程:左侧是有节奏、可点选的自适应对话,右侧是一份从空白逐段长出的真实简历预览。让"从零建简历"从"对着空画布打字"变成"看着自己的简历在对话里成形"。

---

## 2. 核心用户动作(Primary Action)

> **回答 AI 的上下文化追问,看着右侧简历一段段长出来,随时可改、可应用。**

补齐北极星「画布是舞台,对话是旁白」中这个唯一缺舞台的场景。

---

## 3. 设计方向(沿用既定语言)

- 视觉:深色工作台 `#0A0A0A`、sky `#38bdf8` 点缀、少 border;动效轻快贴元素,仅 transform / opacity。
- 气质:锐利、协作、有主见的"搭档"。
- 关键表达:**有人味但可控**——AI 主动、会追问、给得出起点;但用户每一步都看得见进度、能跳过、能直接说。
- 两条红线(anti-goals,已确认):**不变成冗长问卷**、**不退回冷冰冰填表**。

---

## 4. 布局策略

双栏,复用 `LivingCanvas` 的"对话 + 画布"骨架,让 create 模式也启用右栏:

```
┌─ 顶栏:AI 实验室 · <简历名> ················ 实时画布  新对话  关闭 ┐
├──────────────────────────┬────────────────────────────────────┤
│  对话旁白(~42%)           │   简历草稿(~58% · 真实渲染)           │
│  · 顶部 AI 自述计划(动态)   │   · 已写 section 即时落入并 sky 高亮   │
│  · 气泡 + 快捷回复 chips     │   · 未写 section 骨架占位 +"正在长出"  │
│  · 落地日志(可点回溯)       │   · 复用 MagicResumeRenderer         │
│  · 底部 Composer           │                                    │
└──────────────────────────┴────────────────────────────────────┘
```

- **顶部不是固定 `N/5` 进度条**——那是栅栏式填表思维,与"agentic 搭档"冲突。改为 AI **自主声明的动态计划**(对应后端 `agent_plan`:AI 自己决定这次分几步、做什么),用户看到的是"AI 当前在做什么 / 接下来打算做什么",可随对话变化,不被锁死成固定五步。
- 移动端:右栏退为可展开抽屉 / 底部 peek,默认聚焦对话(见 §9 开放项)。

## 5. 关键状态

| 状态 | 用户看到 / 该有的感受 |
|---|---|
| 入场(动态开场) | **开场不预读简历**(隐私,见 §6.5)。AI 温暖地从零开场、动词开头一句话,并**主动提出**"要我参考你现有的简历来给建议吗?"——这是它*请求授权*,不是已经读了。**永不再是固定一句**;开场本身也经过下面的「思考中」与超时兜底 |
| 请求读取授权 | AI 决定要读简历时弹「工具授权」卡:"AI 想读取你的简历来…" + [允许 / 拒绝]。准许后才真正读;拒绝则继续从零、不读(§6.5) |
| 追问中(自适应) | 每轮 2–3 个 chips(从经历里挑 / 我直接说 / 跳过这步)+ 自由输入;答得详细→少追问直接落稿,答得简略→补一问 |
| **思考中(等待首响)** | 发出后**立刻有反馈**:左侧线程尾部出现"思考中"占位(头像 + 三点 / 微光)。**绝不空窗**;首 token 到达即被真正的回复替换。区别于下一行「生成中」——这是首 token *之前*的态。注:**已授权后**读简历才落"正在读取你的简历…"活动行当旁白;未授权不会出现这行 |
| 生成中(首 token 之后) | 右侧目标段 shimmer 微光,左侧打字流,不锁屏 |
| 段落落地 | 右侧该段 sky 一闪→归位;左侧留一条可点回溯日志 |
| 成稿 | 右栏完整草稿 + "应用到简历"(复用现有 `draftReady` banner);可继续追问微调 |
| 空 / 出错 / 超时 | 开场生成失败**或首 token 超时**→回退 i18n 兜底问候(保证永远有得聊、永远不空窗) |

## 6. 交互模型

- 选「引导创建」→ 进双栏 create 模式;**不再 `addAssistant` 写死句**,而是请求一次"开场生成"(见 §7)。
- **开场 = 普通的一轮等待**:开场请求复用与每轮对话*同一条*"等待→流式"路径(同一套"思考中"占位、同一套超时兜底),不为开场单开一套逻辑。开场的"用户输入"是本地构造的种子(目标岗位 + 语言,**不含简历内容**——简历要授权后才进上下文,见 §6.5),**不在线程里显示用户气泡**。
- **等待有旁白,不是空转**:纯思考一个"思考中 + 三点";**已授权后**读简历才把"正在读取你的简历…"作为活动行落在回复*上方*(保留懒创建气泡的次序),让等待"被叙述"。
- chips 点选 = 直接发一条用户消息(复用 `handleSend`),也可无视 chips 自由输入——**chips 是脚手架不是栅栏**。
- AI 回复带 `resume_update / resume_patch` 时右栏即时更新对应段并高亮(`runChat` 已接住)。
- "跳过这步 / 我直接说"等元动作让有经验用户快速通关 = 自适应落地。

## 6.5 简历读取授权(tool-approval · human-in-the-loop)

> 决议(v1.2):AI 读取用户简历**必须先经用户授权**,采用「工具授权弹层」模型——AI 自主决定要不要用工具,要用时**暂停并请示**,准许后才执行。Codex / Claude Code 式,最 agentic、对所有读简历的技能通用(不止 create)。**需 Magic-Core 后端参与**(SSE 单向,授权要能回传并让 agent run 暂停 / 续跑)。

- **原则**:北极星「AI 是合作者(改动先提案、可拒绝、有理由)」延伸到*读*——读也先征得同意。开场、追问全程,凡 agent 想 `read_resume`,先弹卡请示,不偷读;前端也**不再本地把简历摘要塞进种子**(那等于绕过授权把简历喂给 AI)。
- **UX**:思考态里插一张轻量授权卡(非全屏弹窗,贴对话流):一句"AI 想读取你的简历来〈理由〉" + [允许] / [拒绝]。允许→落"正在读取你的简历…"活动行→继续;拒绝→AI 收到拒绝、继续从零、不读。
- **粒度 / 记忆**(实现期定):单次授权 vs 本会话记住("本次对话允许读取");建议"本会话记住"以免反复打断,但首次必问。
- **契约草案见 `design/agent-tool-approval-contract.md`(ADR,待移交 Magic-Core)**——SSE 是单向 server→client,human-in-the-loop 需要后端把 agent run 变成可暂停 / 可续跑,要点:
  - agent 要读 → 发新事件 `tool_approval_request`(`payload:{ toolName:'read_resume', reason, requestId }`),**暂停 run**;
  - 前端渲染授权卡 → 用户决定 → 走一个新的轻量端点(如 `POST /api/chat/approve { runId, requestId, approved }`)回传;
  - 后端据此**续跑或跳过**该工具,继续原 SSE 流(或以同 runId 续流)。
  - 备选(更轻、但不够"自主"):**预授权**——前端在开场前问一次,授权才在请求里带 `allowResumeRead:true`,后端据此决定 `read_resume` 是否可用。省掉暂停 / 续跑,但失去"AI 临场自主请示"的质感。两案取舍留给 Magic-Core ADR。

## 7. 内容与文案(Microcopy)

- **全量走 i18n**:修掉写死开场句,文案入 `locales/{en,zh}`。
- **动态开场**:由 agent 生成(后端 create mode 已支持流式),前端提供"开场种子":目标岗位(若有)+ 语言;配 1 套兜底文案。**种子不含简历内容**——简历要授权后才进上下文(§6.5);开场因此是"温暖从零 + 主动请示读简历",而非"我已经看了你的简历…"。
- 语气沿用 living-canvas §7:**简短、动词开头、不复述用户已见**。
- **等待文案**:用动词进行时,别用"加载中 / 请稍候"。读简历时"正在读取你的简历…";纯思考时一个"思考中"+ 三点足矣,不堆字。
- **超时兜底文案**:首 token 超过阈值(实现期定,建议 ~6–8s)→ 降级到 i18n 兜底开场问候,语气同 §5「入场」的温暖从零版,让用户永远有得聊。
- 骨架占位别用"暂无",用引导式:"说说你最近的项目,这段就补上了"——空状态要教用户,不是报空。

5. **✅ 已交付 · 对话级等待态(问题 3 的修法)**:线程尾部"思考中"占位,由 `runChat`「已发起、未收首 `message_chunk`」驱动;首 chunk / 出错 / 超时即清除。**保留懒创建气泡**(让授权后的 `read_resume` 活动行落在回复上方)。纯前端、零后端,与授权解耦,先行落地。
6. **🟡 需后端 · 简历读取授权(§6.5)**:前端加"工具授权卡"组件 + 授权回传调用;后端加 `tool_approval_request` 事件 + 暂停 / 续跑 + 授权端点(或预授权 flag 的轻量替代)。这是 §8.1 完整开场与一切"agent 读简历"的前置闸门,跨仓,待 Magic-Core ADR。

## 11. 成功判据

- 进入「引导创建」后,用户**看着右侧简历逐段长出来**,而非对着空画布打字。
- 开场**因人而异**(读已有简历 / 目标岗位),不再是固定一句;全量 i18n。
- 全程**不超过问卷式的步数**,有经验用户可一路"跳过 / 直接说"快速成稿,新手有 chips 与进度兜底。
- 任何一轮(含开场)从发送到首字之间**永不空窗**:立刻有"思考中"反馈,读简历时有旁白,超时有兜底。
- 体验有"搭档感",不是填表。
