# AI 实验室 · 活的画布（Living Canvas）— UX 设计稿 v1

> 状态：已定稿（v1，方向已确认）。本稿只定义 UX/交互与状态，不含实现代码。
> 定位：这是 AI 实验室三层进化蓝图里的**地基层**。地基做实后，上层的「智能」（JD⇔简历匹配图、主动教练）与「旅程」（岗位分支、面试官对话）才有附着点。

---

## 0. 背景与北极星

当前 AI 实验室的割裂在于：**智能全活在左侧对话里**（执行卡、报告、分数），而**简历被动地躺在右侧画布**（只读：预览 / Diff / JSON / 评分）。用户是在「跟一个工具说话」，而不是「在简历上和搭档协作」。

北极星：**对着一份"活的"简历，和一个懂你目标的搭档协作。**

本层把两件事合成一个连贯的表面：

- **可对话画布**：简历预览从只读变成可操作——指着任意一处（一条经历、一段摘要、一个标题），就地让 AI 改它。
- **改动评审**：AI 的每一处改动，都是**就地、可逐条接受/拒绝**的提案，并附一句"为什么这么改"。

---

## 1. Feature Summary

把右侧的简历画布从"只读产物"升级为"可协作的工作台"。用户在真实渲染的简历上，hover 或选中任意片段即可让 AI 改写、量化、精简、举证；AI 的改动以就地修订（红删/绿增）呈现，逐条评审后落地。面向正在为某个目标岗位打磨简历的求职者，让"改简历"从"贴 JD → 等一坨结果"变成"指哪打哪、看得见、可掌控"。

---

## 2. 核心用户动作（Primary Action）

> **指着简历里任意一处，让 AI 改它；然后逐条看着接受。**

一切设计为这句话服务。其它能力（整篇优化、翻译、分析）都收敛成"产生一批就地待评审的改动"。

---

## 3. 设计方向（沿用既定语言）

- **气质**：呆萌外表·靠谱内核——温暖、带一点俏皮的搭档语气；对话自然长度不强求简短，建议永远具体、诚实、专业。
- **视觉**：深色工作台（`#0A0A0A`）、sky 点缀、**少 border / 减少割裂**、修订沿用白底简历上的红删(`#dc2626`)/绿增(`#15803d`)。
- **动效**：轻、快、贴着元素（已调校的语言）。
- **核心原则**：
  1. **画布是舞台，对话是旁白。** 工作发生在简历上；对话流退为"入口 + 历史/旁白"。
  2. **能力按需浮现，不做常驻 chrome。** 操作把手 hover 才出现，不污染阅读。
  3. **AI 不是推土机，是合作者。** 任何改动都先提案、可拒绝、有理由。
  4. **就地，而非另开。** 改动标在简历本身上（不再是单独的 Diff tab）。

---

## 4. 布局策略

```
┌─ 顶栏：AI 实验室 · <简历名> ······················· 新对话  关闭 ┐
├──────────────────────────┬────────────────────────────────────┤
│  对话流（旁白 / 历史）      │   简历画布（主舞台 · 可操作）          │
│  · 整篇级请求的入口         │   · 真实渲染的简历                    │
│  · 每次动作留一条紧凑日志    │   · hover/选中 → 浮现操作            │
│  （可收窄成 rail）         │   · 待评审改动就地红删/绿增           │
│                          │                                    │
│                          ├────────────────────────────────────┤
│                          │  评审条（有待评审改动时浮现）          │
│                          │  N 处改动 · ‹上一处 下一处› 全部接受 放弃 │
└──────────────────────────┴────────────────────────────────────┘
```

- **进入"直接操作"时，画布是重心**：对话列可收窄为窄 rail（保留头像 + 历史），把宽度让给简历。
- **评审条**在有待评审改动时从画布底部滑入；无改动时不存在（不占常驻空间）。
- 不引入第三栏 / session 列表（见上一轮结论：与简历版本历史语义重叠、加重割裂）。

## 5. 交互模型（决策核心）

### 决策：元素驱动为主 + 选区驱动为辅

简历是**结构化**内容（section → item → bullet）。因此：

- **主：元素驱动**——hover 一条 bullet / 一段经历 / 一个标题，行首浮出把手，点开是针对"这一条"的快捷动作。结构化、可发现、命中率高。
- **辅：选区驱动**——选中任意文字片段 → 浮出动作条，对"这一段"自由施法。给高级用户更细的控制粒度。

> 备选（已否决）：纯选区驱动——对结构化简历不够直观、发现性差；纯元素驱动——无法处理"只改一句里的半句"。两者合一覆盖最广。

### A. 触发

| 方式 | 触发 | 呈现 |
|---|---|---|
| 元素 hover | 鼠标移到某条目 | 该行**淡高亮** + 行首浮现 `✦` 把手 |
| 元素点击 | 点 `✦` 把手 | 弹出**操作浮层**：快捷动作 chips + 自由指令输入 |
| 选区 | 选中一段文字 | 选区上方浮出**动作条**（优化这段 / 缩短 / 翻译 / 自定义） |
| 对话 | 左侧输入框 | 整篇级请求与技能调用（如"按 JD 优化全篇"）的入口保留 |

### B. 动作（= 作用域受限的技能）

快捷动作按元素类型给：

- **经历 bullet**：量化 · 精简 · 换个更有力的动词 · 补证据
- **个人摘要**：重写 · 调整语气 · 缩短
- **段落标题/排序**：补充这段 · 调整顺序
- **自由指令**：`告诉 AI 怎么改这一条…`（映射到模型，自行决定调用哪个技能）

> 架构含义：技能从"只能整篇"变成**带 `scope`（whole-resume / element / selection）的可定向能力**。详见 §8。

### C. 处理中

- 被作用的片段就地 **shimmer / 微光**；简历其余部分保持可读、可继续操作。
- 不锁全屏、不弹独立 loading。

### D. 提案（待评审改动）

每处 AI 改动 = 一个**就地修订单元**：

```
  ┌ 原文（红删，浅红底，删除线） ─────────────────┐
  │ 负责前端开发工作                               │
  ├ 新版（绿增，浅绿底） ──────────────────────────┤
  │ 主导 3 人前端团队，重构核心模块使首屏提升 40%   │  ✓ 接受  ✕ 放弃  ↻ 再来一版
  └ 理由：补上量化数据，突出影响力 ────────────────┘
```

- 控件簇：`✓ 接受` / `✕ 放弃` / `↻ 再来一版`（重新生成该处）。
- 一句**理由**（hover 可展开更细），让用户知道"为什么"。

### E. 评审（多处时）

当 ≥1 处待评审，画布底部滑入**评审条**：

- `N 处改动待评审` · `‹ 上一处` `下一处 ›`（定位并滚动到改动）· `全部接受` · `全部放弃`
- 键盘：`Tab` 下一处 · `↵` 接受当前 · `Esc` 放弃当前 · `⌘↵` 全部接受。

### F. 落地

- **接受**：改动写入简历（走既有 patch 流程，持久化）→ 该处绿色一闪后归于正常 → 左侧对话留一条紧凑日志（"改写了 工作经历·第 2 条"）。
- **放弃**：干净回退，无残留。
- 整篇级技能（如"按 JD 优化"）= **一次产出一批就地待评审改动** + 评审条接管，而不是另开 Diff tab。

### 对话流的新角色

不再是工作主场，而是：①整篇级请求/技能入口；②**旁白与历史**——每次画布上的动作留一条可点回溯的紧凑日志。把"发生了什么"沉到对话，把"正在做什么"留在画布。

## 6. 关键状态

| 状态 | 用户看到 / 该有的感受 |
|---|---|
| 预览（idle） | 正常渲染的简历；首次使用有一次性 coachmark 提示"可操作" |
| hover 激活 | 当前条目淡高亮 + 行首 `✦`；"这里能动" |
| 选区激活 | 选区上方动作条；"选中的这段能改" |
| 自由指令 | 贴着目标的小输入框；"我可以用大白话说怎么改" |
| 处理中 | 目标片段微光；其余可继续；"它在改这一处，我没被卡住" |
| 待评审（单处） | 就地红删/绿增 + 控件 + 理由；"看得见改了啥、为什么、我能否决" |
| 待评审（多处） | 多处散落标注 + 底部评审条；"我能逐条走查、也能一键全收" |
| 已接受 | 绿色一闪 → 归位 + 对话留痕；"落地了，有记录" |
| 已放弃 | 干净回退；"反悔零成本" |
| 错误 | 目标处错误 chip + `重试`；"哪出错、怎么补救都清楚" |
| 首次/空 | coachmark："试试 hover 任意一条经历，让 AI 帮你改它" |

## 7. 内容与文案（Microcopy）

- 快捷动作：`量化` `精简` `换个动词` `补证据` `重写` `调整语气` `自定义…`
- 自由指令占位：`告诉 AI 怎么改这一条…`
- 理由示例：`补上量化数据` · `动词更有力` · `更突出影响力` · `去掉冗余表述`
- 修订控件：`接受` · `放弃` · `再来一版`
- 评审条：`N 处改动待评审` · `上一处` · `下一处` · `全部接受` · `全部放弃`
- 接受后对话日志：`已改写 · 工作经历 第 2 条`
- coachmark：`hover 任意一条经历，让 AI 就地帮你改`
- 错误：`这处没改成，点一下重试`

> 语气：简短、动词开头、不复述用户已经看到的东西。

## 8. 与现有架构的衔接（交付要点）

1. **技能加 `scope` 概念**：`registry.ts` 里每个 `AiSkill` 增加可作用域（`whole-resume` / `element` / `selection`）。元素/选区动作 = 带 target 的技能调用，产出 → 待评审改动而非另开画布视图。
2. **渲染器要可定位**：`MagicResumeRenderer` 需要给条目/字段输出锚点（如 `data-resume-path="sections.experience[1].description"`），UI 才能把把手与修订**精确贴到**对应节点。这是本层最关键的技术前置。当前描述是 `WysiwygContent` 注入 HTML（bullet 为 `<li>`），需要为可定位粒度补 data 属性。
3. **改动是 patch**：沿用既有 patch 化思路（Core/MCP 已是 JSON Patch）。定义 `PendingChange { id, targetPath, before, after, rationale, status }`；接受 = 应用 patch + 持久化；放弃 = 丢弃。
4. **Diff 视图演进**：从单独的 "Diff" tab → 简历上的**实时待评审态**；保留一个"全部改动"汇总入口即可。
5. **对话承载历史**：复用 `useMessageStore`，每次落地写一条紧凑日志消息（可点回溯）。

## 9. 动效（沿用既定语言，轻快贴元素）

- 把手/操作浮层：`opacity + 6px y` 淡入，~140ms。
- 提案出现：绿增文字在红删下方**展开淡入**（高度用 grid-rows 过渡，避免直接动 height）。
- 接受：绿色一闪（~400ms）→ 归位。
- 评审条：从画布底部滑入；无改动时退出。
- 仅用 transform/opacity；不弹跳、不 elastic。

---

## 10. 边界与开放问题

- **结构化模板（两栏/侧栏）**：把手与修订如何在非线性布局里锚定（依赖 §8.2 的 data-path）。
- **手改与待评审冲突**：用户手动编辑了一处正待评审的内容怎么办（建议：手改即视为放弃该处提案）。
- **超长简历**：评审条"下一处"需滚动定位 + 视口高亮。
- **自由指令 → 技能路由**：大白话指令由谁决定调哪个技能（接真模型即 function-calling；mock 阶段先正则/规则）。
- **整篇技能的量级**：一次几十处改动时，评审条的分组/折叠策略（按 section 分组？）。
- **移动端**：v1 先聚焦桌面工作台；hover 模型在触屏需替代触发（长按/点选），列为 v2。

---

## 11. 落地分期（让它可建）

- **P1（最小闭环）**：元素 hover → 快捷动作 → 单处待评审（就地红删/绿增 + 接受/放弃）。复用我们已做的"diff 注入真实简历"能力。
- **P2**：选区驱动 + 自由指令输入。
- **P3**：多处评审条 + 整篇技能（优化/翻译）改为产出就地待评审改动，替换独立 Diff tab。
- **P4**：上层「智能」挂载——匹配图的缺失项、主动教练的提醒，都锚定到画布的具体条目上。

## 13. 成功判据

- 用户能在**不离开简历**的情况下完成"改某一条"的完整闭环（指 → 改 → 评审 → 落地）。
- 每处改动都**看得见、有理由、可否决**；用户对 AI 的信任与掌控感上升。
- 整篇优化不再是"一坨结果"，而是"一批可逐条走查的就地改动"。
- 上层智能/旅程功能能直接复用本层的"定位 + 待评审"机制，无需重做。
